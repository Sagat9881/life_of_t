## Контекст

Сайт загружает PNG-превью по запросу (lazy). При активной навигации пользователя возможны повторные запросы одного и того же ресурса. Нужен клиентский asset-cache, изолированный в отдельном модуле.

---

## Связанные спецификации

- `docs/specs/technical/visual-docs-site-structure.md` — общая архитектура `site/`.
- `javascript-developer-skill.md` (раздел 6 — управление ресурсами и asset-cache).

---

## Что нужно сделать

1. Реализовать **asset-cache слой** поверх `fetch`/`Image`:
   - Ключ кеша: строка `previewPath`.
   - Значение: `ImageBitmap` (предпочтительно) или `HTMLImageElement`.
   - API: `assetCache.get(previewPath)` → `Promise<ImageBitmap|HTMLImageElement>`.
2. При повторном запросе того же `previewPath` — возвращать из кеша **без сетевого запроса**.
3. Реализовать стратегию вытеснения **LRU** (Least Recently Used):
   - Максимальный размер кеша `N` берётся из конфигурационного объекта (`config.assetCacheSize` или аналог) — **не хардкодить**.
   - При превышении `N` — вытеснять наименее недавно использованный элемент.
4. Состояние загрузки: отображать нейтральный плейсхолдер (CSS-спиннер или серый квадрат через CSS) — **без имён сущностей в стилях или разметке плейсхолдера**.

---

## Ограничения

- Нет хардкода путей, имён сущностей или числа `N` в логике кеша.
- `N` берётся исключительно из конфига.
- Cache-модуль должен быть изолирован — его API не раскрывает внутреннюю структуру LRU вовне.
- Плейсхолдер — нейтральный, без нарративных ссылок (только CSS).

---

## Критерии приёмки

- [ ] Повторный клик по карточке не вызывает сетевого запроса (проверяется через DevTools → Network — запрос отсутствует при повторном вызове).
- [ ] Размер кеша ограничен параметром из конфига; превышение `N` вытесняет LRU-элемент.
- [ ] Нет хардкода путей или имён сущностей в cache-логике (CI-grep-проверка).
- [ ] Плейсхолдер отображается корректно во время загрузки.
- [ ] Cache-модуль покрыт unit-тестом: проверить hit, miss и LRU-вытеснение при `N=2`.
