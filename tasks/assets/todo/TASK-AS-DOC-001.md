## Контекст

Документационный сайт (`site/`) получает данные из двух файлов:
- `site/data/catalog.json` — каталог игровых ассетов, генерируется из `specs-manifest.xml`;
- `site/data/docs.json` — индекс документации, генерируется из `.md`-файлов в `docs/`.

Оба файла создаются bash-скриптами в CI-пайплайне. Алгоритмы скриптов полностью описаны в спецификации `docs/specs/technical/visual-docs-ci-workflow.md`. Запрет хардкода обоснован в ADR-001.

## Что нужно сделать

### 1. Создать `scripts/collect-manifest.sh`

**Назначение:** сгенерировать `site/data/catalog.json` — массив объектов с метаданными каждой не-abstract сущности.

**Алгоритм (псевдокод):**

```
Входные аргументы:
  $1 = <specs-manifest-path>   — путь до specs-manifest.xml
  $2 = <docs-preview-dir>      — корневая директория с preview-данными
  $3 = <output-path>           — путь для записи catalog.json

Шаги:
1. Проверить, что все три аргумента переданы. Если нет — вывести usage и exit 1.
2. Прочитать specs-manifest.xml.
   Для каждого узла <entity> с атрибутом abstract != "true":
     a. Извлечь entityType и entityId.
     b. Сформировать путь: <docs-preview-dir>/<entityType>/<entityId>/card-meta.json
     c. Если файл не найден — вывести stderr: "ERROR: card-meta.json not found for <entityType>/<entityId>" и exit 1.
     d. Прочитать содержимое card-meta.json.
     e. Добавить объект в накапливаемый массив.
3. Записать финальный JSON-массив в <output-path>.
   Перезаписать файл, если он уже существует (идемпотентность).
```

**Инструменты парсинга XML:** `python3 xml.etree.ElementTree` или `xmllint --xpath`.

**Инструменты сборки JSON:** `python3 json` или `jq`.

**Ограничения:**
- Никаких хардкодов имён персонажей, локаций, ассетов — всё только из аргументов и содержимого файлов.
- Пути формируются исключительно из переданных аргументов.

---

### 2. Создать `scripts/collect-docs.sh`

**Назначение:** сгенерировать `site/data/docs.json` — массив объектов-индексов документационных файлов.

**Алгоритм (псевдокод):**

```
Входные аргументы:
  $1 = <docs-dir>    — корневая директория с markdown-документацией
  $2 = <output-path> — путь для записи docs.json

Шаги:
1. Проверить, что оба аргумента переданы. Если нет — вывести usage и exit 1.
2. Найти все .md-файлы:
   find <docs-dir> -name '*.md' -not -path '*/prompts/*'
   (исключить поддиректорию prompts/ — только документация, не промты)
3. Для каждого найденного файла:
   a. Извлечь первую строку вида "# <Заголовок>" — это title.
      Если строка не найдена — использовать имя файла без расширения.
   b. Извлечь slug: имя файла без расширения, приведённое к lowercase-hyphen.
   c. Сформировать contentPath: относительный путь от <docs-dir> до файла.
   d. Собрать объект: { "title": "...", "slug": "...", "contentPath": "..." }
4. Записать JSON-массив объектов в <output-path>.
   Перезаписать, если файл существует (идемпотентность).
```

**Ограничения:**
- Нет хардкода путей или имён файлов в теле скрипта.
- `prompts/` строго исключается через `-not -path '*/prompts/*'`.

---

### 3. Общие требования к обоим скриптам

- **Идемпотентность:** повторный запуск с теми же аргументами всегда перезаписывает выходной файл без ошибок.
- **Fail-fast:** при обнаружении отсутствующего обязательного файла скрипт немедленно завершается с `exit 1` и информативным сообщением в stderr.
- **Shebang:** `#!/usr/bin/env bash`.
- **Исполняемые биты:** `chmod +x` — оба скрипта должны быть исполняемыми в репозитории.
- **Запуск из CI:** скрипты вызываются в CI-шагах `collect-manifest` и `collect-docs` согласно `visual-docs-ci-workflow.md`.

## Критерии приёмки

- [ ] `scripts/collect-manifest.sh` существует и является исполняемым.
- [ ] `scripts/collect-manifest.sh` генерирует корректный `catalog.json` для всех не-abstract сущностей из `specs-manifest.xml`.
- [ ] `scripts/collect-docs.sh` существует и является исполняемым.
- [ ] `scripts/collect-docs.sh` генерирует корректный `docs.json` для всех `.md`-файлов в `docs/`, исключая `prompts/`.
- [ ] Оба скрипта не содержат хардкодов имён персонажей, локаций или ассетов — проверяется code review.
- [ ] При отсутствующем `card-meta.json` скрипт `collect-manifest.sh` завершается с кодом 1 и выводит сообщение об ошибке в stderr.
- [ ] CI-шаг `collect-manifest` завершается успешно после шага `generate-docs-preview`.
- [ ] Повторный запуск обоих скриптов не вызывает ошибок (идемпотентность).
