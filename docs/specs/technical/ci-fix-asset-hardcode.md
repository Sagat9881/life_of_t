## 1. Контекст и цель

ADR-001 устанавливает принцип: **ни код, ни CI-логика не должны содержать явных имён,
идентификаторов или путей нарративных сущностей**. Единственный источник истины — файл
`game-content/life-of-t/src/main/resources/asset-specs/specs-manifest.xml`.

В текущей версии `.github/workflows/asset-generation.yml` выявлены **три шага**, которые прямо
нарушают этот принцип путём перечисления конкретных имён PNG-файлов в статических структурах данных.
Настоящий документ описывает каждое нарушение, его риск и требования к исправлению.

Цель: привести все три шага к data-driven форме, при которой добавление новой сущности
в `specs-manifest.xml` не требует изменения YAML-файла CI.

---

## 2. Описание проблемы

### 2.1. Шаг `Verify generated assets exist`

**Нарушение.** В шаге объявлен bash-массив:

```
EXPECTED=(
  "tanya_idle.png"
  "tanya_walk.png"
  "sam_idle.png"
  "sam_walk.png"
  "bed_static.png"
  "home_room_bg.png"
)
```

Это статический список из шести конкретных имён файлов. Перечисление находится в секции `run:`
шага `Verify generated assets exist` приблизительно на строках 60–68 текущего файла `asset-generation.yml`.

**Риск.** При добавлении новой сущности в `specs-manifest.xml` (например, персонажа `mila`) и
генерации её PNG-ассета — CI **не будет** проверять наличие `mila_idle.png`. Ошибка пройдёт
незамеченной (silent failure): workflow завершится успешно, тест будет «зелёным», но новый
ассет окажется непроверенным. Ошибка обнаружится только при ручном просмотре артефактов.

**Нарушение ADR-001:** раздел 3, п. 3 («CI динамически получает список ожидаемых ассетов…
никакого статического списка в YAML»).

---

### 2.2. Шаг `Validate atlas dimensions`

**Нарушение.** В шаге объявлен Python-словарь:

```
assets = {
  'tanya_idle.png': (192, 48),
  'tanya_walk.png': (256, 48),
  'sam_idle.png':   (96, 20),
  'sam_walk.png':   (144, 20),
  'bed_static.png': (48, 32),
  'home_room_bg.png': (320, 192)
}
```

Словарь одновременно фиксирует: (а) имена файлов для обхода и (б) ожидаемые пиксельные размеры.
Оба поля захардкожены в YAML.

**Риск 1 — пропуск новой сущности.** Аналогичен 2.1: новый ассет не попадает в словарь →
его размеры не проверяются → тихий провал.

**Риск 2 — несоответствие размеров после изменения спека.** Если Java Developer изменит
ожидаемый размер атласа в XML-спеке сущности, но забудет обновить словарь в `asset-generation.yml`,
CI будет проверять устаревшие размеры. Это либо ложноположительный результат (при случайном
совпадении), либо ложноотрицательный (при расхождении).

**Нарушение ADR-001:** раздел 3, п. 2 («Запрещено перечислять конкретные идентификаторы
или имена файлов сущностей в YAML-файлах workflow»).

---

### 2.3. Шаг `Verify no anti-aliasing`

**Нарушение.** В шаге объявлен Python-список:

```
sprites = ['tanya_idle.png', 'tanya_walk.png', 'sam_idle.png', 'sam_walk.png', 'bed_static.png']
```

Список содержит пять конкретных имён и не включает `home_room_bg.png`, уже присутствующий
в предыдущих шагах, — то есть даже в рамках текущего хардкода он неполный.

**Риск.** Новый ассет не попадёт в проверку на отсутствие антиалиасинга. Если генератор
случайно применит сглаживание к новому спрайту, CI не выявит дефект.

**Нарушение ADR-001:** раздел 3, п. 2 — те же основания, что в 2.1.

## 3. Требования к исправлению

### FR-1. Шаг `Verify generated assets exist` — динамический список

Список ожидаемых файлов должен строиться **в runtime шага** путём парсинга `specs-manifest.xml`.

Требования к поведению шага:
1. Шаг читает `specs-manifest.xml` из пути
   `game-content/life-of-t/src/main/resources/asset-specs/specs-manifest.xml`.
2. Из манифеста извлекаются все элементы `<entity>` с атрибутом `abstract="false"`.
3. Для каждой такой сущности вычисляется ожидаемое имя PNG-файла по шаблону именования
   (см. раздел 4).
4. Для каждого ожидаемого имени файла шаг проверяет его наличие в `$ASSET_OUTPUT_DIR`.
5. Если хотя бы один ожидаемый файл отсутствует — шаг завершается с ненулевым кодом возврата.
6. Шаг выводит в лог: суммарное количество ожидаемых файлов, список найденных, список отсутствующих.

**Не допускается:** наличие в YAML или в inline-скрипте шага любых строк вида `tanya_*`,
`sam_*`, `bed_*`, `mila_*` или иных конкретных идентификаторов сущностей.

---

### FR-2. Шаг `Validate atlas dimensions` — размеры из метаданных ассета

Ожидаемые размеры PNG-файла должны читаться **не из YAML**, а из метаданных, связанных
с каждой сущностью.

Требования к поведению шага:
1. Шаг итерирует по тем же сущностям `abstract="false"` из `specs-manifest.xml`.
2. Для каждой сущности ожидаемые ширина и высота PNG берутся из **`card-meta.json`** — файла
   метаданных ассета, расположенного рядом с выходным PNG в `$ASSET_OUTPUT_DIR`.
3. Поля в `card-meta.json`, из которых читаются размеры:
   - `expectedWidth` — ожидаемая ширина в пикселях,
   - `expectedHeight` — ожидаемая высота в пикселях.
4. Если `card-meta.json` для данной сущности отсутствует — шаг **выводит предупреждение** и
   пропускает проверку размеров для этой сущности (не падает с ошибкой). Факт пропуска
   фиксируется в выводе лога.
5. Если размеры не совпадают с фактическими PNG-размерами — шаг завершается с ненулевым кодом.

> **Открытый вопрос для Java Developer (TASK-TEST-DOC-001):**
> Подтвердить, что `asset-generator` создаёт `card-meta.json` для каждого PNG в режиме
> стандартной генерации, и что поля `expectedWidth` / `expectedHeight` присутствуют в этом файле.
> Если нет — Java Developer должен добавить их генерацию (задача в рамках TASK-TEST-DOC-001
> или как отдельная подзадача).

---

### FR-3. Шаг `Verify no anti-aliasing` — динамический список из манифеста

Требования к поведению шага:
1. Шаг получает список PNG-файлов для проверки тем же способом, что FR-1: парсинг манифеста,
   фильтрация `abstract="false"`, формирование имён по шаблону именования (раздел 4).
2. Для каждого файла из списка шаг проверяет, что файл существует и имеет ненулевой размер.
3. Дополнительная проверка (если применимо): шаг может расширяться будущими правилами
   проверки качества пикселей, но всегда опирается на динамический список — не на
   статический перечень.
4. При отсутствии любого ожидаемого файла шаг завершается ошибкой.

---

### NFR-1. Единый источник списка сущностей

Все три исправленных шага обязаны использовать **один и тот же** способ получения списка
сущностей: парсинг `specs-manifest.xml`. Дублирование логики парсинга в трёх отдельных
inline-скриптах нежелательно. Рекомендуется вынести парсинг в отдельный Python-скрипт
в `scripts/` и вызывать его из каждого шага (это согласуется с TASK-TEST-DOC-002,
которая предусматривает реализацию `scripts/collect-manifest.sh`).

---

### NFR-2. Отсутствие конкретных идентификаторов сущностей в YAML

После рефакторинга `asset-generation.yml` не должен содержать строк вида:
`tanya`, `sam`, `bed`, `home_room`, `mila` — и любых других конкретных идентификаторов
нарративных сущностей. Критерий проверяется командой:

```
grep -E '(tanya|sam|bed|mila|home_room)' .github/workflows/asset-generation.yml
```

Команда должна вернуть пустой результат (исключая комментарии).

---

### NFR-3. Валидация манифеста как предварительный шаг

Перед шагом `Verify generated assets exist` должен существовать отдельный шаг валидации
XML-манифеста. Если `specs-manifest.xml` недоступен или повреждён — CI должен завершаться
с явной ошибкой и понятным сообщением, а не с неочевидным сбоем в середине шага проверки.

---

## 4. Шаблон именования PNG-файлов

Шаблон зафиксирован на основании `visual-docs-preview-mode.md` (раздел FR-2, поле `spriteAtlasFile`).

### 4.1. Правило формирования имени

Для каждой сущности с `abstract="false"` в `specs-manifest.xml` имя PNG-файла idle-анимации
формируется по следующему **детерминированному** правилу:

```
{entity_id}_{animation_name}.png
```

Где:
- `entity_id` — последний сегмент значения атрибута `path` элемента `<entity>` в манифесте.
  Пример: если `path="characters/tanya"`, то `entity_id = "tanya"`.
- `animation_name` — имя основной (idle) анимации. Для всех персонажей по умолчанию — `idle`.
  Для фоновых объектов и мебели — `static`.

### 4.2. Правила для разных типов сущностей

| Тип сущности (`path`-префикс) | `animation_name` по умолчанию | Пример имени PNG    |
|-------------------------------|-------------------------------|----------------------|
| `characters/*`                | `idle`                        | `tanya_idle.png`     |
| `furniture/*`                 | `static`                      | `bed_static.png`     |
| `locations/*`                 | `bg`                          | `home_room_bg.png`   |
| `ui/*`                        | `default`                     | `hud_default.png`    |

> **Примечание.** Если XML-спек сущности содержит явное поле `spriteAtlasFile`
> (в `card-meta.json` или в `docs-preview.json`), это значение имеет приоритет над
> шаблонным именем. Логика парсинга должна проверять это поле первым.

### 4.3. Соответствие шаблону именования в `visual-docs-preview-mode.md`

`visual-docs-preview-mode.md` (FR-2) определяет поле `spriteAtlasFile` как:
«имя PNG idle-анимации», где `id` — последний сегмент `path`. Таким образом, шаблон
именования в этом документе и в `visual-docs-preview-mode.md` **консистентны**.

## 5. Алгоритм динамического построения списка (пошагово)

Алгоритм предназначен для реализации в bash-шаге CI с использованием
`python3 xml.etree.ElementTree`. Внешние зависимости не требуются.

### Шаг 1 — Проверка доступности манифеста

Убедиться, что файл `specs-manifest.xml` существует по ожидаемому пути
(`game-content/life-of-t/src/main/resources/asset-specs/specs-manifest.xml`).
Если файл отсутствует — завершить шаг с ненулевым кодом и явным сообщением об ошибке.

### Шаг 2 — Парсинг манифеста

Прочитать XML-файл с помощью стандартного парсера (`python3 xml.etree.ElementTree`
без внешних зависимостей, или `xmllint` при наличии). Получить дерево элементов.

### Шаг 3 — Фильтрация сущностей

Из всех элементов с тегом `<entity>` отобрать только те, у которых:
- атрибут `abstract` имеет значение `"false"` (строковое сравнение).

Сущности с `abstract="true"` (базовые типы) пропускаются полностью — они не порождают PNG.

### Шаг 4 — Формирование ожидаемого имени PNG

Для каждой отфильтрованной сущности:
1. Извлечь значение атрибута `path`.
2. Разбить по символу `/` и взять последний сегмент — это `entity_id`.
3. Определить тип сущности по первому сегменту `path` (characters / furniture / locations / ui).
4. Сопоставить тип с таблицей `animation_name` из раздела 4.2.
5. Сформировать итоговое имя файла: `{entity_id}_{animation_name}.png`.

### Шаг 5 — Проверка существования файла

Для каждого сформированного имени файла проверить наличие файла
`$ASSET_OUTPUT_DIR/{expected_filename}`. Вести два счётчика:
`found_count` и `missing_count`. Добавлять отсутствующие файлы в список `missing`.

### Шаг 6 — Вывод результата и завершение

Вывести в лог:
- суммарное количество ожидаемых файлов,
- список найденных (с размерами),
- список отсутствующих.

Если `missing_count > 0` — завершить с кодом `exit 1`.
Если все файлы найдены — завершить с кодом `exit 0`.

## 7. Задачи для реализации

### TASK-TEST-DOC-001 — Рефакторинг `.github/workflows/asset-generation.yml`

| Поле               | Значение |
|--------------------|----------|
| **ID**             | TASK-TEST-DOC-001 |
| **Тип**            | testing / CI |
| **Связанная спека**| `docs/specs/technical/ci-fix-asset-hardcode.md` (настоящий документ) |
| **ADR**            | ADR-001 |

**Описание:** Заменить все захардкоженные списки имён PNG в шагах `Verify generated assets exist`,
`Validate atlas dimensions`, `Verify no anti-aliasing` на data-driven логику, описанную
в разделах FR-1, FR-2, FR-3 и алгоритме (раздел 5). Убедиться, что YAML-файл не содержит
конкретных идентификаторов сущностей (NFR-2). Добавить предварительный шаг валидации
манифеста (NFR-3).

**Критерии приёмки:**
- `grep -E '(tanya|sam|bed|home_room)' .github/workflows/asset-generation.yml` возвращает
  пустой результат.
- При добавлении новой `abstract="false"` сущности в манифест CI автоматически проверяет
  её ассет без изменений YAML.
- Все три исправленных шага завершаются успешно на текущем наборе ассетов.
- Шаг валидации манифеста добавлен и выполняется до шагов верификации ассетов.
