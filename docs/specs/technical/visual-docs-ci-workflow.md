## 1. Контекст и цель

Необходим новый GitHub Actions workflow `.github/workflows/docs-site.yml`, обеспечивающий автоматическую сборку и публикацию статического документационного сайта «Life of T» на GitHub Pages.

Workflow должен:

- реагировать на изменения XML-спек, документации и скриптов без ручного вмешательства;
- запускать генератор ассетов в режиме `docs-preview` (см. `visual-docs-preview-mode.md`);
- собирать машиночитаемые JSON-каталоги через bash-скрипты;
- собирать и деплоить статический сайт.

**Ключевое ограничение согласно ADR-001**: workflow, bash-скрипты и конфигурации сайта **не содержат хардкода имён, идентификаторов или путей нарративных сущностей**. Единственный источник перечня сущностей — `specs-manifest.xml`.

---

## 2. Функциональные требования

### FR-1 — Автоматические триггеры

Workflow запускается при push в ветки `main` и `develop` при изменении любого из следующих путей:

| Паттерн триггера                   | Причина                                          |
|------------------------------------|--------------------------------------------------|
| `game-content/**/asset-specs/**`   | Изменение XML-спек сущностей (включая манифест) |
| `docs/visual-specs/**`             | Изменение документационных спек                 |
| `scripts/**`                       | Изменение bash-скриптов сборки каталогов        |
| `site/src/**`                      | Изменение исходников сайта                      |

Дополнительно поддерживается ручной запуск через `workflow_dispatch` без входных параметров.

### FR-2 — Data-driven исполнение

Ни один шаг workflow не содержит явного перечисления имён сущностей, персонажей, локаций или идентификаторов ассетов. Все данные о составе сущностей получаются динамически из `specs-manifest.xml`.

### FR-3 — Атомарность провала

Если любой из шагов завершается с ненулевым кодом возврата (кроме шагов, помеченных как `continue-on-error`), workflow немедленно прекращает выполнение. GitHub Pages не получает частично собранный артефакт.

## 3. Шаги workflow

Ниже приведены все шаги в порядке исполнения. Каждый шаг выполняется в рамках одного job `build-and-deploy` на runner `ubuntu-latest`.

| # | Шаг                   | Action / Команда                                              | Вход                                                 | Выход                       | Условие провала                              |
|---|-----------------------|---------------------------------------------------------------|------------------------------------------------------|-----------------------------|----------------------------------------------|
| 1 | `checkout`            | `actions/checkout@v4`                                         | —                                                    | Полный checkout репозитория | Недоступен репозиторий                       |
| 2 | `setup-java`          | `actions/setup-java@v4` (Java 21, Temurin, кэш Maven)         | —                                                    | JDK 21 в PATH               | Недоступна дистрибуция Temurin               |
| 3 | `build-generator`     | `mvn -B install -DskipTests -f asset-generator/pom.xml`       | `asset-generator/pom.xml` и исходники                | Исполняемый JAR генератора  | Ошибка компиляции или packaging              |
| 4 | `generate-docs-preview` | `mvn -B ... -Dassets.output-mode=docs-preview -Dassets.output-dir=target/docs-preview/` | `specs-manifest.xml`, XML-спеки сущностей в `game-content/**/asset-specs/` | `target/docs-preview/docs-preview.json` | Генератор завершился с кодом ≠ 0; файл `docs-preview.json` не создан |
| 5 | `collect-manifest`    | `bash scripts/collect-manifest.sh <manifest-path> <docs-preview-dir>` | `specs-manifest.xml`, `target/docs-preview/`          | `site/data/catalog.json`    | Скрипт завершился с кодом ≠ 0; файл не создан или пустой массив |
| 6 | `collect-docs`        | `bash scripts/collect-docs.sh <docs-dir>`                      | `docs/`                                              | `site/data/docs.json`       | Скрипт завершился с кодом ≠ 0               |
| 7 | `validate-catalog`    | `bash scripts/validate-catalog.sh <manifest-path> <catalog-path>` | `specs-manifest.xml`, `site/data/catalog.json`        | Статус pass / fail (exit 0 / 1) | Для хотя бы одной не-abstract сущности из манифеста нет записи в каталоге |
| 8 | `build-site`          | `npm ci && npm run build` в директории `site/`                 | `site/src/`, `site/data/catalog.json`, `site/data/docs.json` | `site/dist/`           | Ошибка сборки npm                            |
| 9 | `upload-pages`        | `actions/upload-pages-artifact@v3` (path: `site/dist/`)        | `site/dist/`                                         | Загруженный артефакт Pages  | Артефакт не загружен                         |
| 10 | `deploy-pages`       | `actions/deploy-pages@v4`                                      | Артефакт из шага 9                                   | URL GitHub Pages            | Деплой отклонён GitHub Pages API             |

> **Примечание по шагам 5–7**: конкретные имена bash-скриптов и точные пути к аргументам передаются переменными окружения, объявленными в секции `env:` workflow — чтобы избежать дублирования путей в теле шагов.

## 4. Спецификация `scripts/collect-manifest.sh`

### 4.1. Назначение

Скрипт собирает `catalog.json` — главный источник данных для документационного сайта. Выходной файл описан в `visual-docs-site-structure.md`.

### 4.2. Входные параметры

| Параметр   | Позиция | Описание                                                   |
|------------|---------|------------------------------------------------------------|
| `MANIFEST` | `$1`    | Абсолютный или относительный путь к `specs-manifest.xml`   |
| `PREVIEW_DIR` | `$2` | Путь к директории с выходом генератора (`target/docs-preview/`) |

Скрипт **не использует** никаких других путей, кроме переданных аргументами. Пути не захардкожены.

### 4.3. Выход

Файл `site/data/catalog.json` — JSON-массив объектов. Каждый объект соответствует одной не-abstract сущности из манифеста. Структура объекта каталога определена в `visual-docs-site-structure.md` и соответствует формату `EntityDocsDescriptor` из `visual-docs-preview-mode.md` (раздел FR-2).

### 4.4. Алгоритм (словами, без кода)

1. Принять аргументы `MANIFEST` и `PREVIEW_DIR`; завершиться с кодом 1, если любой из них не передан или путь не существует.
2. Распарсить XML-файл `MANIFEST` стандартными средствами (например, `python3 xml.etree.ElementTree` или `xmllint`).
3. Извлечь список элементов `<entity>`, у которых атрибут `abstract` равен `"false"` (или отсутствует, если это означает не-abstract). Для каждого элемента получить значение атрибута `path`.
4. Для каждого `path` сформировать ожидаемый путь к файлу дескриптора внутри `PREVIEW_DIR`. Конкретная структура файлов внутри `PREVIEW_DIR` определяется Java Developer в рамках `visual-docs-preview-mode.md`.
5. Прочитать дескриптор каждой сущности. Если файл для конкретного `path` отсутствует — вывести предупреждение в stderr и пропустить сущность (не прерывать обработку).
6. Собрать массив всех прочитанных дескрипторов в единую JSON-структуру.
7. Записать итоговый JSON-массив в `site/data/catalog.json`, создав директорию при необходимости.
8. Вывести итоговый счётчик: сколько сущностей обработано, сколько пропущено.
9. Завершиться с кодом 0, если массив не пустой; с кодом 1, если ни одна сущность не обработана.

### 4.5. Ограничения (ADR-001)

- **Запрещено** перечислять конкретные имена или идентификаторы сущностей в теле скрипта.
- **Запрещено** использовать фиксированные пути кроме тех, что переданы аргументами.
- **Запрещено** содержать логику вида `if path == "characters/tanya"`.

## 5. Спецификация `scripts/collect-docs.sh`

### 5.1. Назначение

Скрипт обходит директорию `docs/` и собирает `docs.json` — список разделов документации для навигации на сайте.

### 5.2. Входные параметры

| Параметр   | Позиция | Описание                          |
|------------|---------|-----------------------------------|
| `DOCS_DIR` | `$1`    | Путь к директории `docs/`         |

### 5.3. Выход

Файл `site/data/docs.json` — JSON-массив объектов. Каждый объект описывает один Markdown-файл из `docs/`. Структура объекта:

| Поле          | Тип    | Описание                                                                |
|---------------|--------|-------------------------------------------------------------------------|
| `title`       | string | Первый заголовок H1 (`# ...`) из файла; если заголовок отсутствует — имя файла без расширения |
| `slug`        | string | Имя файла без расширения и без пути директории                          |
| `contentPath` | string | Относительный путь к файлу от корня репозитория                         |

### 5.4. Алгоритм (словами, без кода)

1. Принять аргумент `DOCS_DIR`; завершиться с кодом 1, если путь не передан или не существует.
2. Найти все файлы с расширением `.md` рекурсивно в `DOCS_DIR` (эквивалент `find DOCS_DIR -name '*.md'`).
3. Для каждого найденного файла:
   a. Прочитать содержимое файла.
   b. Найти первую строку, начинающуюся с `# ` (один символ `#` и пробел). Если найдена — извлечь текст заголовка без символа `#` и пробела. Если не найдена — использовать имя файла без расширения.
   c. Сформировать `slug` из имени файла без расширения.
   d. Сформировать `contentPath` как путь к файлу относительно корня репозитория.
   e. Создать объект `{ title, slug, contentPath }`.
4. Собрать массив всех объектов.
5. Записать итоговый JSON-массив в `site/data/docs.json`, создав директорию при необходимости.
6. Завершиться с кодом 0.

## 6. Спецификация шага `validate-catalog` (динамическая валидация)

### 6.1. Назначение

Шаг `validate-catalog` обеспечивает, что при добавлении новой сущности в `specs-manifest.xml` её запись автоматически появляется в `catalog.json`. Это машиночитаемая верификация соответствия между манифестом и собранным каталогом.

### 6.2. Входные параметры скрипта `scripts/validate-catalog.sh`

| Параметр       | Позиция | Описание                                              |
|----------------|---------|-------------------------------------------------------|
| `MANIFEST`     | `$1`    | Путь к `specs-manifest.xml`                           |
| `CATALOG`      | `$2`    | Путь к `site/data/catalog.json`                       |

### 6.3. Алгоритм (словами, без кода)

1. Принять аргументы `MANIFEST` и `CATALOG`; завершиться с кодом 1, если любой из них не передан или не существует.
2. Распарсить `MANIFEST`. Извлечь список `path` всех элементов `<entity abstract="false"/>`.
3. Прочитать `CATALOG`. Извлечь список значений поля `path` (или `id`) всех объектов в массиве.
4. Для каждого `path` из манифеста проверить, что в каталоге присутствует объект с соответствующим `path` (или соответствующим `id`, если `id` является последним сегментом `path`).
5. Если для любой сущности из манифеста запись в каталоге не найдена — вывести имя отсутствующей сущности в stderr с пометкой `::error::` (GitHub Actions annotation) и завершиться с кодом 1.
6. Если все сущности присутствуют — вывести счётчик проверенных сущностей и завершиться с кодом 0.

### 6.4. Ограничения (ADR-001)

- Скрипт **не содержит** ожидаемого списка сущностей — он всегда получает его динамически из манифеста.
- Добавление новой сущности в `specs-manifest.xml` автоматически включает её в проверку без изменения скрипта или workflow.

| ID     | Требование                                                                                      |

| NFR-5  | Workflow не зависит от внешних реестров или приватных npm-пакетов кроме стандартных Actions      |
