## 1. Контекст и цель

### 1.1. Назначение

Документационный сайт — **чисто статическое** приложение, публикуемое через GitHub Pages. Источником
данных служат два JSON-артефакта, собранные bash-скриптами в CI:

1. `site/data/catalog.json` — каталог всех не-abstract сущностей; формируется из `specs-manifest.xml`
   и `card-meta.json` скриптом `scripts/collect-manifest.sh`.
2. `site/data/docs.json` — структура технической документации; формируется из `docs/*.md`
   скриптом `scripts/collect-docs.sh`.

Никакой серверной части нет. Сайт не содержит хардкода идентификаторов, имён, путей
или типов конкретных нарративных сущностей — согласно
[ADR-001](../decisions/ADR-001-visual-docs-data-independence.md).

### 1.2. Связь с ADR-001

> **Принцип независимости данных (ADR-001, §3.3):** JavaScript-клиент не хранит перечней
> персонажей, локаций или ассетов в коде. Все списки сущностей поступают через JSON-каталог,
> собранный `scripts/collect-manifest.sh`. Разрешено использовать типовые ключи
> (`characters`, `locations` и т. п.) для обращения к полям дескрипторов, но не конкретные
> идентификаторы экземпляров.

## 2. Архитектура сайта

### 2.1. Технологический стек

**Выбор: чистый HTML / CSS / Vanilla JS без SSG-фреймворков.**

| Критерий            | Vanilla HTML/JS | Astro / 11ty            |
|---------------------|-----------------|-------------------------|
| Зависимости         | Ноль            | Node.js, пакеты npm     |
| Сложность сборки    | Нет (копирование файлов) | Требуется `npm build` |
| Поддерживаемость    | Максимальная    | Зависит от версий       |
| Динамические данные | fetch + DOM API | Частично статический    |

**Обоснование:** каталог сущностей динамически формируется в CI и меняется при каждом добавлении
новой сущности. SSG требует пересборки шаблонов при изменении данных; Vanilla JS читает
`catalog.json` при загрузке и строит DOM итерационно — без пересборки и без зависимостей.
Это соответствует принципу минимума зависимостей (`javascript-developer-skill.md`, §5).

Разрешена CSS-утилита (например, Tailwind CDN или минимальный reset.css) — без JS-зависимостей.

### 2.2. Структура директорий

```
site/
├── src/                    # Исходники сайта (HTML, CSS, JS)
│   ├── index.html          # Точка входа; содержит пустые контейнеры для DOM
│   ├── css/
│   │   └── style.css       # Минимальные стили сетки и компонентов
│   └── js/
│       ├── main.js         # Инициализация: fetch catalog.json → render
│       ├── catalog.js      # Модуль: разбор и хранение catalog.json
│       ├── docs.js         # Модуль: разбор и хранение docs.json
│       ├── components/
│       │   ├── entity-card.js      # Компонент EntityCard
│       │   ├── animation-preview.js # Компонент AnimationPreview
│       │   ├── spec-panel.js       # Компонент SpecPanel
│       │   ├── filter-bar.js       # Компонент FilterBar
│       │   └── doc-section.js      # Компонент DocSection
│       └── events.js       # Центральный диспетчер событий
├── data/                   # JSON-данные (копируются CI, не редактируются вручную)
│   ├── catalog.json        # Каталог сущностей (из scripts/collect-manifest.sh)
│   └── docs.json           # Структура документации (из scripts/collect-docs.sh)
├── public/                 # Статические ассеты (PNG спрайтов, шрифты, иконки)
│   └── assets/             # Спрайт-атласы сущностей (копируются CI)
└── dist/                   # Артефакт сборки для публикации (GitHub Pages)
```

**Правило**: файлы в `site/data/` и `site/public/assets/` **никогда не редактируются вручную** —
только CI-скриптами. Исходники сайта находятся в `site/src/`.

### 2.3. Принцип инициализации

При загрузке страницы `main.js` выполняет:

1. `fetch('./data/catalog.json')` — загрузка каталога.
2. `fetch('./data/docs.json')` — загрузка структуры документации.
3. Передача данных в `catalog.js` и `docs.js` для хранения.
4. Итерационный обход каталога → создание элементов `EntityCard` для каждой записи.
5. Инициализация `FilterBar` по уникальным значениям `entityType` из каталога.
6. Инициализация навигации `DocSection` по разделам из `docs.json`.

**Запрещено** производить какие-либо операции на основе конкретных значений `entityId`
или `entityType` в коде инициализации (`javascript-developer-skill.md`, §3.1).

## 3. Компоненты страницы

Каждый компонент описан через **данные, которые он потребляет**, и **поведение, которое он реализует**.
Вёрстка не специфицируется — это зона ответственности JavaScript Developer.

### 3.1. EntityCard

**Данные:**

| Поле            | Тип     | Источник         | Описание                                   |
|-----------------|---------|------------------|--------------------------------------------|
| `entityId`      | string  | `catalog.json`   | Уникальный идентификатор сущности          |
| `entityType`    | string  | `catalog.json`   | Тип: одно из значений каталога             |
| `caption`       | string  | `catalog.json`   | Отображаемое имя                           |
| `previewPath`   | string\|null | `catalog.json` | Путь к PNG первой анимации (относительно `site/public/`) |
| `animationCount`| number  | `catalog.json`   | Количество анимаций сущности               |
| `specVersion`   | string  | `catalog.json`   | Версия спецификации                        |

**Поведение:**

- Карточка создаётся для каждой записи каталога итерационно — без явного перечисления сущностей в коде.
- Если `previewPath === null` — карточка отображает placeholder-иконку, не ломая отрисовку.
- **Одиночный клик** по карточке → вызов обработчика, привязанного к `entityId`, который
  инициирует открытие `AnimationPreview` для данной сущности.
- **Двойной клик** по карточке → вызов обработчика, который инициирует открытие `SpecPanel`
  для данной сущности.
- Карточка принимает состояние `active` / `hidden` от `FilterBar` на основе соответствия
  текущему фильтру по `entityType`.

### 3.2. AnimationPreview

**Данные:**

| Поле          | Тип          | Источник                        | Описание                               |
|---------------|--------------|---------------------------------|----------------------------------------|
| `entityId`    | string       | Событие от EntityCard           | Идентификатор сущности для загрузки    |
| `previewPath` | string\|null | `catalog.json` → запись по `entityId` | Путь к PNG спрайт-атласа          |

**Поведение:**

- Компонент **не загружает PNG при загрузке страницы** — только по событию клика на карточке
  (lazy load).
- При получении события с `entityId`: находит в каталоге запись, читает `previewPath`,
  загружает PNG по этому пути.
- Если `previewPath === null` — показывает placeholder-сообщение, не падает.
- Показывает первую анимацию сущности (sprite-atlas.png, описанный в `card-meta.json` → `previewPath`).
- При открытии другой карточки — заменяет содержимое на данные новой сущности.

### 3.3. SpecPanel

**Данные:**

| Поле       | Тип    | Источник                        | Описание                                    |
|------------|--------|---------------------------------|---------------------------------------------|
| `entityId` | string | Событие от EntityCard           | Идентификатор сущности                      |
| `specPath` | string | `catalog.json` → запись по `entityId` | Путь к markdown-файлу спецификации сущности |

**Поведение:**

- Открывается по двойному клику на карточке **или** по нажатию отдельной кнопки на карточке.
- При открытии: загружает markdown-файл по `specPath` (fetch), конвертирует в HTML
  (допустимо использование минимальной библиотеки, например `marked.js`), отображает как текст.
- Панель раскрываемая (expand/collapse) — не модальное окно.
- При повторном открытии той же сущности — не перезагружает файл (кешируется в памяти).
- Закрывается по повторному двойному клику или кнопке закрытия.

### 3.4. FilterBar

**Данные:**

| Поле            | Тип           | Источник       | Описание                                          |
|-----------------|---------------|----------------|---------------------------------------------------|
| Список типов    | string[]      | `catalog.json` | Уникальные значения `entityType` по всему каталогу |
| Активный фильтр | string\|null  | Внутреннее состояние | Текущий выбранный тип или null (все сущности) |

**Поведение:**

- При инициализации **итерационно** строит список фильтров по уникальным значениям `entityType`
  из `catalog.json`. Никаких хардкод-значений вида `["characters", "locations", ...]` в коде нет
  ([ADR-001, §3.3](../decisions/ADR-001-visual-docs-data-independence.md)).
- Клик по фильтру → устанавливает активный фильтр → скрывает карточки, у которых
  `entityType !== activeFilter`.
- Повторный клик по активному фильтру → сбрасывает фильтр (показываются все карточки).
- Специальный элемент «Все» всегда присутствует и сбрасывает фильтр.

### 3.5. DocSection

**Данные:**

| Поле          | Тип    | Источник     | Описание                                   |
|---------------|--------|--------------|--------------------------------------------|
| `title`       | string | `docs.json`  | Заголовок раздела                          |
| `slug`        | string | `docs.json`  | URL-идентификатор раздела                  |
| `content`     | string | `docs.json`  | Содержимое раздела (HTML или markdown)     |
| `contentPath` | string | `docs.json`  | Путь к внешнему файлу (если content пустой) |

**Поведение:**

- Навигация по разделам технической документации формируется итерационно из `docs.json`.
- Клик по разделу → отображает содержимое (из `content` или загружает `contentPath`).
- Активный раздел подсвечен в навигации.
- Навигация не содержит хардкоженных заголовков разделов в HTML.

## 4. Контракт данных

### 4.1. Формат `catalog.json`

Файл — JSON-массив объектов. Каждый объект — одна не-abstract сущность из `specs-manifest.xml`.

```jsonc
{
  "schemaVersion": "1.0",
  "generatedAt": "2026-04-03T12:00:00Z",
  "entities": [
    {
      "entityId": "tanya",
      "entityType": "characters",
      "caption": "Tanya",
      "previewPath": "assets/tanya/idle/sprite-atlas.png",
      "specPath": "docs/visual-specs/characters/tanya/visual-specs.md",
      "animationCount": 3,
      "layerCount": 4,
      "specVersion": "1.2.0"
    }
  ]
}
```

**Описание полей верхнего уровня:**

| Поле             | Тип    | Обязательно | Описание                                              |
|------------------|--------|-------------|-------------------------------------------------------|
| `schemaVersion`  | string | Да          | Версия схемы файла (семвер)                           |
| `generatedAt`    | string | Да          | ISO 8601 timestamp генерации                          |
| `entities`       | array  | Да          | Массив дескрипторов сущностей                         |

**Описание полей каждой записи `entities[]`:**

| Поле            | Тип          | Обязательно | Описание                                                                       |
|-----------------|--------------|-------------|--------------------------------------------------------------------------------|
| `entityId`      | string       | Да          | Уникальный идентификатор; последний сегмент пути из манифеста (`tanya`, `sofa`) |
| `entityType`    | string       | Да          | Тип сущности; первый сегмент пути (`characters`, `locations`, `furniture`, `ui`) |
| `caption`       | string       | Да          | Отображаемое имя (первая буква заглавная)                                      |
| `previewPath`   | string\|null | Да          | Относительный путь к PNG первой анимации (от корня `site/`); null если PNG нет  |
| `specPath`      | string\|null | Да          | Относительный путь к markdown-спецификации сущности; null если спека не найдена |
| `animationCount`| number       | Да          | Количество анимаций в XML-спеке сущности (0 если анимации отсутствуют)         |
| `layerCount`    | number       | Да          | Количество слоёв в XML-спеке сущности (0 если слои отсутствуют)               |
| `specVersion`   | string       | Да          | Версия XML-спецификации из атрибута `version` в XML-спеке; `"0.0.0"` если отсутствует |

**Инварианты:**

- `entityId` уникален в пределах массива.
- `entityType` берётся из данных (`catalog.json`), никогда не хардкодится в коде сайта.
- Записи с `abstract: true` в манифесте **не включаются** в `catalog.json`.

### 4.2. Формат `docs.json`

Файл — JSON-объект с массивом разделов документации.

```jsonc
{
  "schemaVersion": "1.0",
  "generatedAt": "2026-04-03T12:00:00Z",
  "sections": [
    {
      "title": "Архитектура",
      "slug": "architecture",
      "content": "",
      "contentPath": "docs/specs/technical/visual-docs-site-structure.md"
    },
    {
      "title": "Решения",
      "slug": "decisions",
      "content": "## ADR-001\n\nПринцип независимости данных...",
      "contentPath": null
    }
  ]
}
```

**Описание полей верхнего уровня:**

| Поле            | Тип    | Обязательно | Описание                                |
|-----------------|--------|-------------|-----------------------------------------|
| `schemaVersion` | string | Да          | Версия схемы файла                      |
| `generatedAt`   | string | Да          | ISO 8601 timestamp генерации            |
| `sections`      | array  | Да          | Массив разделов документации            |

**Описание полей каждой записи `sections[]`:**

| Поле          | Тип          | Обязательно | Описание                                                                                   |
|---------------|--------------|-------------|--------------------------------------------------------------------------------------------|
| `title`       | string       | Да          | Заголовок раздела для навигации                                                            |
| `slug`        | string       | Да          | URL-идентификатор раздела (строчный латинский, дефисы); уникален в пределах массива        |
| `content`     | string       | Да          | Содержимое раздела в markdown (может быть пустой строкой, если задан `contentPath`)       |
| `contentPath` | string\|null | Да          | Относительный путь к внешнему `.md`-файлу; null если `content` содержит данные напрямую   |

**Инварианты:**

- Ровно одно из `content` (непустая строка) или `contentPath` (не null) должно быть задано.
- `slug` уникален в пределах массива.

## 5. Правила JavaScript Developer

Данный раздел является обязательным для выполнения. Ссылки на
`javascript-developer-skill.md`.

### 5.1. Запрет хардкода (§3.1, §3.3 skill-файла; ADR-001 §3.3)

- **Запрещено** использовать в коде строковые литералы вида `"tanya"`, `"idle_walk"`,
  `"sofa_brown"` или любые другие конкретные идентификаторы сущностей.
- **Запрещено** создавать `switch-case` / `if-else` по значениям `entityId`.
- **Запрещено** создавать `enum` или `const`-объекты с перечислением идентификаторов сущностей.
- **Запрещено** хардкодить список типов сущностей в `FilterBar`; типы берутся итерационно
  из `catalog.json`.

### 5.2. Итерационное построение DOM (§2, §8 skill-файла)

- `FilterBar` строит список фильтров через `Set` уникальных значений `entityType`
  из массива `entities`, не через статический список.
- Сетка карточек строится итерационным обходом `catalog.json → entities[]`.

### 5.3. Абстрактные обработчики событий (§3.1 skill-файла)

- Обработчики событий (клик, двойной клик) привязываются к `entityId` из данных,
  а не к конкретному DOM-узлу.
- Логика выбора компонента для открытия (`AnimationPreview` или `SpecPanel`) основана
  на типе события (одиночный / двойной клик), а не на `entityId`.

### 5.4. Запрет DOM/CSS-фреймворков (§3.2 skill-файла)

- Запрещены React, Vue, Angular и аналогичные SPA-фреймворки.
- Допустимы: Vanilla JS + минимальный CSS, `marked.js` для рендера markdown в SpecPanel.
- DOM используется как инструмент отображения данных, а не как хранилище состояния.

### 5.5. Lazy-загрузка ресурсов (§6 skill-файла)

- PNG спрайтов загружаются только при клике на карточку (`AnimationPreview`), не при
  загрузке страницы.
- Markdown-спецификации загружаются только при открытии `SpecPanel`.
- Повторные запросы к одному ресурсу кешируются в памяти (объект/Map в `catalog.js`).

## 6. Сценарии использования

### UC-1 — Загрузка сайта

```
1. Пользователь открывает docs-сайт (GitHub Pages URL).
2. Браузер загружает index.html → main.js.
3. main.js: fetch('./data/catalog.json') → Promise<catalogData>.
4. main.js: fetch('./data/docs.json') → Promise<docsData>.
5. catalog.js сохраняет catalogData.entities в памяти.
6. docs.js сохраняет docsData.sections в памяти.
7. entity-card.js итерационно создаёт DOM-элемент для каждой записи entities[].
8. filter-bar.js строит список фильтров по уникальным entityType из entities[].
9. doc-section.js строит навигацию по sections[].
10. Сетка карточек отображается. FilterBar активен. Навигация по документации доступна.
```

### UC-2 — Фильтрация по типу

```
1. Пользователь кликает по фильтру (значение — любое entityType из catalog.json).
2. filter-bar.js устанавливает activeFilter = выбранный entityType.
3. Итерационный обход всех карточек:
   - карточки с entityType === activeFilter → состояние visible;
   - остальные → состояние hidden.
4. Пользователь видит только карточки выбранного типа.
5. Клик по «Все» → сброс фильтра → все карточки visible.
```

### UC-3 — Просмотр анимации (одиночный клик)

```
1. Пользователь кликает по карточке с entityId = X.
2. events.js диспатчит событие {type: "card:click", entityId: X}.
3. animation-preview.js получает событие, ищет в catalogData запись с entityId = X.
4. Читает previewPath из найденной записи.
5. Если previewPath !== null: fetch(previewPath) → загружает PNG.
6. AnimationPreview отображает спрайт-атлас PNG.
7. Если previewPath === null: отображает placeholder-сообщение.
```

### UC-4 — Открытие спецификации (двойной клик)

```
1. Пользователь двойным кликом открывает карточку с entityId = Y.
2. events.js диспатчит событие {type: "card:dblclick", entityId: Y}.
3. spec-panel.js получает событие, ищет в catalogData запись с entityId = Y.
4. Читает specPath из найденной записи.
5. Если specPath !== null и файл не кеширован: fetch(specPath) → загружает markdown.
6. Markdown конвертируется в HTML и отображается в SpecPanel.
7. Если specPath === null: панель показывает сообщение «Спецификация не найдена».
8. Повторный двойной клик или кнопка «Закрыть» → SpecPanel сворачивается.
```

## 7. Нефункциональные требования

| ID     | Требование                                                                                                 |

| NFR-1  | Сайт работает без серверной части — открывается через `file://` или `npx serve`.                          |
