## Контекст

В рамках разработки визуального документационного сайта «Life of T» возникла системная архитектурная проблема:
существующий CI-пайплайн `.github/workflows/asset-generation.yml` содержит **захардкоженный список** ожидаемых PNG-файлов
(`tanya_idle.png`, `sam_idle.png`, `tanya_walk.png`, `sam_walk.png`, `bed_static.png`, `home_room_bg.png`) и их конкретных размеров.

Единственный источник истины о составе сущностей — `game-content/life-of-t/src/main/resources/asset-specs/specs-manifest.xml`,
который содержит 12 персонажей, 12 локаций, 18 предметов и 3 UI-группы.

Любой новый персонаж или локация, добавленные в манифест, **не отражаются** автоматически в:
- верификации CI (список `EXPECTED` в bash-шаге),
- шаге валидации атласных размеров (`Validate atlas dimensions`),
- шаге проверки анти-алиасинга (`Verify no anti-aliasing`),
- логике сборки документационного сайта.

Это создаёт нарушение принципа **Data Independence** — фундаментального архитектурного принципа проекта.

---

## Проблема

> Код, CI-логика и сайт содержат явные ссылки на конкретные имена сущностей.
> Добавление новой сущности требует изменений в нескольких местах вместо одного (`specs-manifest.xml`).

Конкретные нарушения в `asset-generation.yml`:

1. **Bash-массив `EXPECTED`** — хардкод имён PNG-файлов в шаге `Verify generated assets exist`.
2. **Python-словарь `assets`** — хардкод пар `(имя файла → ожидаемые размеры)` в шаге `Validate atlas dimensions`.
3. **Python-список `sprites`** — хардкод имён файлов в шаге `Verify no anti-aliasing`.
4. **Условие запуска приложения** — проверка `[ -f "${{ env.ASSET_OUTPUT_DIR }}/tanya_idle.png" ]`
   как сигнала об окончании генерации (также захардкожено).

---

## Варианты решения

### Вариант A — Сохранить хардкод, добавить комментарий "обновлять вручную"

- Самый простой.
- **Отклонён:** противоречит архитектурному принципу проекта. При добавлении новой сущности разработчик
  обязан помнить о ручном обновлении CI. Это ошибка проектирования.

### Вариант B — Читать список сущностей из `specs-manifest.xml` в CI через bash/python/xmllint

- CI сам парсит манифест и динамически строит список ожидаемых файлов.
- Не нужно новых инструментов — только стандартный `xmllint` или Python 3 с `xml.etree`.
- Валидация CI становится **data-driven**: сколько сущностей в манифесте — столько и проверяется PNG.
- **Принято.**

### Вариант C — Генератор ассетов публикует манифест выходных артефактов в JSON

- Генератор после работы сохраняет `sprite-atlas.json` или `generated-assets-manifest.json`.
- CI читает этот JSON и проверяет только то, что было действительно сгенерировано.
- Более гибко, но требует изменений в Java-коде генератора.
- **Принято как дополнение к варианту B** — генератор уже производит `sprite-atlas.json`,
  CI должен использовать его для верификации вместо захардкоженного списка.

---

## Принятое решение

**Применить комбинацию вариантов B и C:**

1. **В CI** (`asset-generation.yml`) заменить все захардкоженные списки файлов на динамическую генерацию
   из `specs-manifest.xml` (парсинг атрибута `path` у элементов `<entity abstract="false"/>`).

2. **Условие ожидания завершения** генерации заменить на проверку наличия `sprite-atlas.json`
   (или счётчика файлов ≥ N, где N читается из манифеста).

3. **Для документационного сайта** — сайт никогда не содержит список персонажей в своём коде.
   Данные поступают только через JSON-артефакты, собранные bash-скриптами из манифеста.

4. **Для bash-скриптов** (`scripts/`) — все скрипты сбора данных для сайта итерируют по
   `specs-manifest.xml` и не содержат имён сущностей напрямую.

5. **Для генератора ассетов** (`asset-generator/`) — добавить output-mode `docs-preview`,
   который дополнительно генерирует JSON-дескриптор каждой сущности для сайта.
   Список сущностей генератор получает исключительно из манифеста.

---

## Последствия

### Положительные

- Добавление новой сущности = добавление строки в `specs-manifest.xml` + создание XML-спека.
  CI и сайт обновятся автоматически без изменений кода.
- Нет риска «забыть обновить CI» при добавлении нового персонажа.
- Документационный сайт всегда отражает актуальный состав сущностей.

### Отрицательные / риски

- CI-скрипты усложняются: необходим парсинг XML.
  Смягчение: используется стандартный `python3 xml.etree` — без внешних зависимостей.
- Если манифест повреждён, CI упадёт с неочевидной ошибкой.
  Смягчение: добавить отдельный шаг валидации манифеста перед генерацией.
- Генератор должен корректно обрабатывать `abstract="true"` и не генерировать для них ассеты.
  Это уже реализовано согласно контексту, но должно быть покрыто тестом.

---

## Критерии соответствия

Решение считается реализованным, если:

- [ ] В `asset-generation.yml` нет ни одного захардкоженного имени PNG-файла или имени сущности.
- [ ] Список ожидаемых ассетов в CI строится динамически из `specs-manifest.xml`.
- [ ] Добавление новой `<entity>` в манифест без изменений CI приводит к корректному прохождению CI.
- [ ] Документационный сайт рендерит новую сущность без изменений HTML/JS/CSS кода.
- [ ] В `scripts/` нет файлов с захардкоженными именами сущностей.
