# ADR-001 — Принцип независимости данных для docs-site и CI-пайплайна

| Поле           | Значение                                                                 |
|----------------|--------------------------------------------------------------------------|
| **Статус**     | Accepted                                                                 |
| **Дата**       | 2026-04-03                                                               |
| **Автор**      | System Analyst                                                           |
| **Затрагивает**| CI/CD, asset-generator, docs-site, scripts/, Java Developer, JS Developer |
| **Связанные артефакты** | `game-content/life-of-t/src/main/resources/asset-specs/specs-manifest.xml`, `.github/workflows/asset-generation.yml`, `docs/prompts/_core/unified-asset-schema.xml` |

---

## 1. Контекст

Проект «Life of T» переходит к генерации документационного сайта из тех же XML-спецификаций, которые
используются для генерации игровых ассетов. Единственным источником истины о составе нарративных сущностей
(персонажи, локации, предметы, UI-группы) является
`game-content/life-of-t/src/main/resources/asset-specs/specs-manifest.xml`.

**Обнаруженное нарушение.** В текущей версии `.github/workflows/asset-generation.yml` присутствует
несколько мест, где конкретные имена PNG-файлов перечислены статически:

- bash-массив `EXPECTED` в шаге `Verify generated assets exist`,
- python-словарь `assets` в шаге `Validate atlas dimensions`,
- python-список `sprites` в шаге `Verify no anti-aliasing`,
- условие ожидания запуска приложения (проверка конкретного файла как сигнала о готовности).

Это прямое нарушение главного архитектурного принципа проекта: **ни код, ни CI-логика, ни логика сайта
не должны зависеть от конкретных имён, идентификаторов или путей нарративных сущностей**.

**Риски текущего состояния:**

1. При добавлении новой сущности в `specs-manifest.xml` CI не проверит её ассет автоматически —
   потребуется ручное изменение YAML.
2. Если изменение YAML забыто, новый ассет не будет верифицирован, и ошибка пройдёт незамеченной
   (silent failure).
3. Документационный сайт не сможет автоматически отражать новые сущности без правки шаблонов/кода.

---

## 2. Статус

**Accepted** — настоящее решение обязательно для всех ролей и всех артефактов фичи docs-site
начиная с 2026-04-03.

---

## 3. Решение

### 3.1. Общие правила (для всех ролей)

1. **Единственный источник списка сущностей** — `specs-manifest.xml`.
   Любой код, скрипт или конфигурация, которым нужен перечень персонажей, локаций, предметов или UI-групп,
   обязан читать его из манифеста (или из JSON-каталога, сгенерированного из манифеста скриптом
   `scripts/collect-manifest.sh`).

2. **Запрещено** перечислять конкретные идентификаторы или имена файлов сущностей в:
   - YAML-файлах workflow (`.github/workflows/`),
   - JavaScript-коде клиента (`application/`),
   - Java-коде бэкенда (`backend/`) и генератора ассетов (`asset-generator/`),
   - bash/python-скриптах (`scripts/`),
   - конфигурационных файлах документационного сайта.

3. **CI динамически** получает список ожидаемых ассетов из `specs-manifest.xml` путём парсинга
   атрибута `path` у элементов `<entity abstract="false"/>`. Никакого статического списка в YAML.

4. **Документационные спецификации** хранятся исключительно в `docs/visual-specs/`.
   Директория `game-content/` предназначена только для игровых ассетов и никогда не содержит
   docs-specific файлов.

5. **Допустимое исключение**: системные типы — `characters`, `locations`, `furniture`, `ui` —
   могут присутствовать как enum-константы в Java-коде и как ключи в JS-конфигах.
   Конкретные идентификаторы экземпляров этих типов использовать запрещено.

---

### 3.2. Правила для Java Developer

- Генератор ассетов (`ru.lifegame.assets`) обязан получать список сущностей для обработки
  исключительно из манифеста через агрегат/репозиторий `SpecsManifestRepository`.
- Запрещено объявлять `List<String>` или `Map<String, ...>` с именами конкретных сущностей
  в Java-классах — только динамическая загрузка из XML/JSON.
- Для docs-site режима генератор реализует output-mode `docs-preview`, который формирует
  JSON-дескриптор каждой сущности. Список сущностей берётся из манифеста, не из кода.

### 3.3. Правила для JavaScript Developer

- Клиент (`application/`) никогда не содержит захардкоженный перечень персонажей, локаций или ассетов.
- Все списки сущностей для рендера на docs-сайте поступают через API-эндпоинт или через
  JSON-каталог, собранный `scripts/collect-manifest.sh`.
- Разрешено использовать типовые ключи (`characters`, `locations` и т. п.) для обращения к полям
  дескрипторов — но не конкретные идентификаторы сущностей.

### 3.4. Правила для CI-автора (workflow)

- Все шаги верификации ассетов в `.github/workflows/asset-generation.yml` формируют
  список ожидаемых файлов динамически, парсингом `specs-manifest.xml`
  (стандартными инструментами: `python3 xml.etree`, `xmllint`).
- Условие ожидания завершения генерации использует `sprite-atlas.json` или счётчик файлов,
  рассчитанный из манифеста, — но не имя конкретного PNG-файла.
- При добавлении нового шага верификации разработчик обязан проверить, что шаг data-driven.

## 4. Последствия

### 4.1. Положительные

- Добавление новой сущности сводится к одному изменению: добавить строку в `specs-manifest.xml`
  и создать XML-спек. CI и сайт подхватывают её автоматически.
- Исключается класс «забытых» обновлений CI — ни одна новая сущность не останется без верификации.
- Документационный сайт всегда отражает актуальный состав сущностей без правки HTML/JS/CSS кода.

### 4.2. Требуемый рефакторинг

Для приведения проекта в соответствие с настоящим решением необходимо выполнить следующие задачи:

| ID задачи          | Артефакт                              | Суть изменения                                                      |
|--------------------|---------------------------------------|---------------------------------------------------------------------|
| TASK-TEST-DOC-001  | `.github/workflows/asset-generation.yml` | Заменить все захардкоженные списки на динамический парсинг манифеста |
| TASK-TEST-DOC-002  | `scripts/collect-manifest.sh`         | Реализовать скрипт сбора данных из `specs-manifest.xml` в JSON      |
| TASK-BE-DOC-001    | `ru.lifegame.assets` (asset-generator) | Убедиться, что все списки сущностей читаются через `SpecsManifestRepository` |
| TASK-FE-DOC-001    | `application/` (JS-клиент)            | Убедиться, что docs-сайт не содержит перечней сущностей в коде      |

### 4.3. Риски реализации

- CI-скрипты усложняются за счёт XML-парсинга.
  Смягчение: использовать `python3 xml.etree` без внешних зависимостей.
- Если `specs-manifest.xml` повреждён или недоступен, CI упадёт с неочевидной ошибкой.
  Смягчение: добавить отдельный шаг валидации манифеста перед генерацией ассетов.

## 5. Альтернативы, которые были отклонены

### Альтернатива A — Сохранить хардкод и добавить комментарий «обновлять вручную»

**Отклонена.** Это текущее состояние (`asset-generation.yml`). Хардкод создаёт структурную
зависимость кода от данных, которая нарушается при любом изменении состава сущностей.
Риск silent failure неприемлем.

### Альтернатива B — Генерировать отдельный `docs-manifest.xml` параллельно с `specs-manifest.xml`

**Отклонена.** Создаёт два конкурирующих источника истины, которые могут расходиться.
Принцип Single Source of Truth запрещает дублирование манифестов.
