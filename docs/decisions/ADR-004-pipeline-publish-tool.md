# ADR-004 — Pipeline Publish Tool

## Статус
Accepted — 2026-04-02

## Контекст

Пайплайн `build-status-dashboard` требует публикации статического HTML-дашборда
на GitHub Pages после каждого запуска. Необходимо выбрать инструмент деплоя
и формат скриптов сбора метрик.

## Решение

### Инструмент публикации

**Выбран: `peaceiris/actions-gh-pages@v3`**

Аргументы:
- Зрелый action с ~4k stars, широко используется в open-source проектах.
- Простая конфигурация: `publish_dir` + `publish_branch`.
- Не требует отдельной настройки Git-идентификации — использует `GITHUB_TOKEN`.
- Поддерживает `commit_message` с `[skip ci]` для предотвращения цепочки запусков.

### Формат скриптов

**Выбран: bash (`.sh`)**

Аргументы:
- Доступен на `ubuntu-latest` без дополнительной установки.
- Инструменты `find`, `grep`, `wc`, `jq` предустановлены.
- Минимальные зависимости, нет Docker/JVM overhead для простых подсчётов.
- Скрипты переиспользуемы локально разработчиком без CI.

## Отклонённые альтернативы

| Альтернатива | Причина отказа |
|---|---|
| `JamesIves/github-pages-deploy-action` | Требует дополнительной настройки токена |
| Python-скрипты | Избыточность для задач `find/grep/wc` |
| GitHub Actions native `gh` CLI | Меньше контроля над выходными переменными |

## Последствия

- Все скрипты должны использовать `#!/usr/bin/env bash` и `set -euo pipefail`.
- Скрипты принимают аргументы для указания директорий (тестируемость локально).
- `continue-on-error: true` — только на нестабильных шагах (fetch-ci-status, asset-generator).
- `permissions: contents: write` — исключительно на job `deploy`.
