## Задача

Создай **Architecture Decision Record (ADR-001)** в `docs/decisions/ADR-001-visual-docs-data-independence.md`.

### Контекст решения

Проект переходит к генерации документационного сайта из тех же XML-спек, что используются для игровых ассетов. Перед началом реализации необходимо зафиксировать архитектурное решение, которое будет обязательным для всех ролей и всех артефактов этой фичи.

Это решение вытекает из **главного принципа архитектуры Life of T**: ни код, ни CI-логика, ни логика сайта **не должны зависеть от конкретных имён, идентификаторов или путей** нарративных сущностей, персонажей, локаций и ассетов.

Обнаруженное текущее нарушение: в `.github/workflows/asset-generation.yml` список ожидаемых PNG (`tanya_idle.png`, `sam_idle.png`, …) захардкожен напрямую в YAML — это прямое нарушение принципа. При добавлении нового персонажа workflow упадёт или, что хуже, не проверит новый ассет молча.

### Что должно быть в ADR

Структура по стандарту ADR:

1. **Заголовок**: ADR-001 — Принцип независимости данных для docs-site и CI-пайплайна.
2. **Статус**: Accepted.
3. **Контекст**: опиши ситуацию — переход к docs-site, обнаруженное нарушение в CI, риски хардкода.
4. **Решение**: сформулируй правила:
   - Список сущностей в коде, CI и логике сайта **читается только** из `specs-manifest.xml` (или из JSON-каталога, сгенерированного из него скриптом).
   - Никакого перечисления имён персонажей/локаций/ассетов в YAML, JavaScript, Java или bash-скриптах.
   - CI динамически получает список ожидаемых ассетов из манифеста, не из хардкода.
   - Документационные спеки хранятся в отдельном namespace (`docs/visual-specs/`), никогда в `game-content/`.
   - Исключение: имена системных типов (`characters`, `locations`, `furniture`, `ui`) допустимы как enum-константы, но не конкретные идентификаторы сущностей.
5. **Последствия**:
   - При добавлении новой сущности в `specs-manifest.xml` CI и сайт обновляются автоматически без изменения кода.
   - Требует рефакторинга `asset-generation.yml` (задача TASK-TEST-DOC-001).
   - Требует реализации `scripts/collect-manifest.sh`.
6. **Альтернативы, которые были отклонены**: хардкод списка в YAML (текущее состояние); генерация отдельного `docs-manifest.xml` параллельно с `specs-manifest.xml` (дублирование источника истины).
7. **Связанные артефакты**: `game-content/life-of-t/src/main/resources/asset-specs/specs-manifest.xml`, `.github/workflows/asset-generation.yml`, `docs/prompts/_core/unified-asset-schema.xml`.

### Критерии готовности

- [ ] ADR создан в `docs/decisions/ADR-001-visual-docs-data-independence.md`.
- [ ] В ADR явно сформулированы правила для каждой роли (Java Developer, JS Developer, CI-автор).
- [ ] В ADR указан список артефактов, которые должны быть обновлены для соблюдения принципа.
- [ ] ADR не содержит конкретных имён персонажей или локаций в качестве примеров обязательного поведения.

> **Ограничение:** не пиши код, не меняй workflow-файлы — только фиксируй решение.
