## Задача

Создай техническую спецификацию в `docs/specs/technical/ci-fix-asset-hardcode.md`.

### Контекст

В `.github/workflows/asset-generation.yml` обнаружены следующие нарушения принципа независимости данных (ADR-001):

1. **Шаг `Verify generated assets exist`**: захардкожен массив `EXPECTED=("tanya_idle.png" "tanya_walk.png" "sam_idle.png" ...)`. При добавлении персонажа список нужно обновлять вручную — иначе новый ассет не проверяется.
2. **Шаг `Validate atlas dimensions`**: захардкожен словарь имя_файла → ожидаемый размер. Нарушение то же.
3. **Шаг `Verify no anti-aliasing`**: захардкожен список спрайтов.

Все три шага нарушают ADR-001 и подлежат исправлению.

### Что должно быть в техспеке

1. **Описание проблемы**: точные строки в `asset-generation.yml`, которые нарушают принцип, с объяснением риска (пример: новый персонаж добавлен в манифест, но workflow не проверит его ассет → тихий провал).

2. **Требования к исправлению**:
   - Шаг `Verify generated assets exist`: список ожидаемых файлов должен **динамически** строиться из `specs-manifest.xml` — читать все не-abstract entity, формировать ожидаемые имена файлов по шаблону (шаблон именования определяется в `visual-docs-preview-mode.md`).
   - Шаг `Validate atlas dimensions`: ожидаемые размеры берутся из `card-meta.json` каждого ассета (поле из спеки), а не из хардкода.
   - Шаг `Verify no anti-aliasing`: список берётся из тех же динамических данных.

3. **Алгоритм динамического построения списка** (словами для bash-шага):
   - Распарсить `specs-manifest.xml` (xmllint или python3 xml.etree).
   - Для каждого `<entity abstract="false">` извлечь путь.
   - Сформировать ожидаемое имя PNG по детерминированному шаблону (шаблон зафиксировать в спеке).
   - Проверить существование файла в `$ASSET_OUTPUT_DIR`.

4. **Связанные артефакты**: ADR-001, `visual-docs-preview-mode.md`, `specs-manifest.xml`.
5. **Задачи для реализации**: TASK-TEST-DOC-001 (рефакторинг существующего workflow).
